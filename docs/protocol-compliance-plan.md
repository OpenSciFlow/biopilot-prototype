# Protocol compliance plan

BioPilot should be a reference implementation for a narrow OpenSciFlow path, not a general autonomous agent.

The first compliance target is:

```text
mdanalysis-trajectory-analysis + molecular-dynamics-stability-analysis + run_manifest.json
```

## Compliance stages

| Stage | Evidence | Required before |
|---|---|---|
| Manifest check | Plugin manifest validates and command templates pass guardrails | Any execution |
| Workflow check | Workflow schema, DAG, plugin list, and artifact handoff validate | Workflow selection |
| Environment check | Local readiness report records package availability | Dry run |
| Dry run | Plugin dry-run command exits with `0` in a documented environment | R3 claim |
| Smoke test | Tiny public sample produces expected files | R4 claim |
| Run record | `run_manifest.json` validates against schema | R5 claim |
| Report check | Report includes citations, limitations, and artifact list | R6-style demo claim |

## Runner boundaries

The BioPilot runner must:

- load a workflow template from a reviewed local path;
- load plugin manifests referenced by the workflow;
- refuse unknown workflow names;
- refuse missing required inputs;
- refuse plugin manifests below the required readiness level for the requested action;
- render only reviewed command templates;
- keep resolved paths inside the approved run directory or approved input paths;
- write logs and run records even when execution fails after startup.

The runner must not:

- execute arbitrary shell text generated by an LLM;
- auto-install packages or download model weights without explicit approval;
- upload user inputs by default;
- claim biological binding, drug efficacy, clinical meaning, or experimental validation.

## Minimal runner flow

1. Parse user task.
2. Select `molecular-dynamics-stability-analysis`.
3. Validate the workflow template.
4. Load required plugin manifests.
5. Validate plugin manifests and command templates.
6. Check local environment readiness.
7. Check required input files and hashes.
8. Render reviewed command templates.
9. Execute steps inside a run directory.
10. Register artifacts and logs.
11. Validate `run_manifest.json`.
12. Render report with citations and limitations.

## First compliance demo bar

The first public demo is acceptable only when:

- MDAnalysis dry-run passes and evidence is committed;
- the selected sample dataset has license, citation, size, and hash metadata;
- the workflow artifact handoff validator passes;
- the generated run record validates;
- the report repeats the scientific limitations from the manifest and workflow template.
